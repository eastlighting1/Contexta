# Contexta 테스트 가이드

이 가이드는 현재 프로토타입 단계에서 `Contexta` 워크플로를 검증하는 방법을 설명합니다.

핵심 아이디어는 간단합니다:

- 시맨틱(Semantics)을 먼저 테스트하세요.
- 변경 사항을 입증하는 가장 작은 테스트 스위트를 사용하세요.
- 예제는 설명뿐만 아니라 실제 실행과 연결되어 있어야 합니다.

## 현재 테스트 러너 (Test Runner)

프로젝트는 `pytest`를 사용합니다.

저장소 기반의 테스트 실행의 경우, `pytest`는 프로젝트 설정을 통해 이미 `src/` 경로를 전달받으므로 `PYTHONPATH`를 수동으로 설정하지 않고도 일반적인 명령들이 작동합니다.

이는 로컬 체크아웃 환경에서의 임시 스크립트와는 다른 점이며, 임시 스크립트의 경우 여전히 `PYTHONPATH=src`를 사용하는 것이 현재로서 가장 안전한 경로입니다.

## 유용한 명령들

### 전체 테스트 수행

```powershell
uv run pytest -q
```

대규모 머지나 릴리스 단계 전에 가장 폭넓은 신뢰성을 확보하고 싶을 때 사용하세요.

### 핵심 엔드 투 엔드(E2E) 흐름

```powershell
uv run pytest tests/e2e/test_capture_to_report.py -q
```

다음 사항에 영향을 주는 변경 사항이 있을 때 사용하세요:

- 핵심 온보딩
- 쿼리/리포트 동작
- 퍼사드 레벨의 읽기 워크플로

### 퀵스타트 예제 검증

```powershell
uv run pytest tests/e2e/test_quickstart_examples.py -q
```

다음 사항에 영향을 주는 변경 사항이 있을 때 사용하세요:

- `README.md` 퀵스타트 안내
- `docs/user-guide/getting-started.md`
- `examples/quickstart/`
- 신규 사용자를 위한 공용 온보딩 경로

### 복구 예제 검증

```powershell
uv run pytest tests/e2e/test_recovery_examples.py -q
```

다음 사항에 영향을 주는 변경 사항이 있을 때 사용하세요:

- `examples/recovery/`
- 백업, 재생 또는 아티팩트 전송 예제
- 운영자 대상의 복구 온보딩

## 현재 검증 범위

저장소에서 오늘날 가장 강력한 워크플로 수준의 증거는 다음에서 나옵니다:

- `tests/e2e/test_capture_to_report.py`
- `tests/e2e/test_quickstart_examples.py`
- `tests/e2e/test_recovery_examples.py`

이들을 통해 다음 내용을 다룹니다:

- 퍼사드 생명주기 및 읽기 흐름
- 쿼리, 비교, 진단 및 리포트 동작
- 퀵스타트 예제 검증
- 복구 예제 검증

## 계층별 테스트 모델

문서 및 설계 기준점에서는 테스트 스토리를 계층별로 설명합니다.

### 유닛 (Unit)

다음 항목들에 유닛 테스트를 사용하세요:

- 헬퍼 함수
- 직렬화기(serializers) 및 역직렬화기(deserializers)
- 소규모 검증 및 파싱 규칙

### 계약 (Contract)

표준 모델 검증, 결정론적 직렬화 및 안정적인 결과 형태를 위해 계약 테스트를 사용하세요.

### 플레인 통합 (Plane Integration)

다음 항목들에 플레인 통합 테스트를 사용하세요:

- 메타데이터 스토어 동작
- 레코드 추가(append) 및 재생
- 아티팩트 수집(ingest) 및 검증

### 복구 (Recovery)

다음 항목들에 복구 및 마이그레이션 테스트를 사용하세요:

- 재생 동작
- 백업 및 복구

### 인터페이스 (Surface)

다음 항목들에 인터페이스 테스트를 사용하세요:

- Python 퍼사드 동작
- CLI 동작
- HTTP JSON 동작
- HTML UI 동작

### 엔드 투 엔드 (End-To-End)

사용자 여정이 여러 계층에서 한꺼번에 여전히 잘 작동하는지 신뢰를 얻고 싶을 때 엔드 투 엔드 테스트를 사용하세요.

## 문서 변경 사항을 검증하는 방법

문서는 항상 실행 가능한 현실과 연결되어 있어야 합니다.

다음에 변화가 있다면:

- 시작하기(getting-started) 예제
  - 온보딩 스크립트나 가장 가까운 E2E 흐름을 다시 실행하세요.
- 일반적인 쿼리 또는 리포트 가이드
  - 핵심 E2E 흐름을 다시 실행하세요.
- 복구 가이드
  - 복구 예제 커버리지나 가장 가까운 복구 테스트 스위트를 다시 실행하세요.

문서 위주의 변경 시, 목표는 모든 것을 맹목적으로 다시 실행하는 것이 아닙니다. 가이드 내용이 여전히 사실임을 입증하는 가장 가까운 증거를 다시 실행하는 것이 목표입니다.

## 무엇을 단언(Assert)할 것인가

서식에 민감한 단언보다는 시맨틱(semantic) 단언을 선호하세요.

좋은 단언 예시:

- 실행 스냅샷에 기대되는 실행 ID, 스테이지 및 레코드가 포함되어 있음
- 비교 결과에 기대되는 메트릭 또는 스테이지 차이가 노출됨
- 리포트에 기대되는 제목과 섹션이 있음

약한 단언 예시:

- 계약의 일부가 아닌 부수적인 정렬 순서
- 여전히 빠르게 진화 중인 출력물에 대한 크고 깨지기 쉬운 문자열 스냅샷

## 예제 검증 기대 사항

공용 문서의 예제는 다음 중 최소 하나를 입증해야 합니다:

- 표준 임포트 경로가 작동함
- 워크스페이스가 생성되고 읽힐 수 있음
- 실행 결과가 쿼리될 수 있음
- 리포트가 빌드될 수 있음

예제는 내부 모듈이나 비공개 헬퍼 경로에 암시적으로 의존해서는 안 됩니다.

## 프로토타입 참고 사항

현재 프로토타입 단계에서:

- 소스 트리 스크립트 스토리는 여전히 `PYTHONPATH=src`에 의존합니다.
- 패키지 및 CLI 이름은 `contexta`로 정렬되었습니다.

이것이 테스트 가이드에서 세련된 '어디서나 설치 및 실행 가능'한 스토리보다 실행 가능한 저장소 명령과 파일 기반 테스트 스위트를 강조하는 이유입니다.
