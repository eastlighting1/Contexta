# Contexta 핵심 개념

이 페이지는 사용자 가이드의 나머지 내용을 이해하기 위해 필요한 핵심 개념들을 소개합니다.

`Contexta`를 사용하기 위해 모든 내부 세부 사항을 알 필요는 없지만, 다음에 대해서는 명확한 멘탈 모델(mental model)이 필요합니다:

- 워크스페이스(workspace)
- 실행(run) 및 스테이지(stage)
- 레코드(record) 및 아티팩트(artifact)
- 리니지(lineage) 및 프로버넌스(provenance)
- 쿼리(query) 및 리포트(report)
- 완전성(completeness) 및 기능 저하(degraded states)

## 핵심 루프 (The Core Loop)

제품은 로컬 우선(local-first)의 핵심 루프를 기반으로 구축되었습니다:

1. 캡처 (capture)
2. 저장 (store)
3. 쿼리 (query)
4. 리포트 (report)

이것이 시스템을 통과하는 가장 짧은 개념적 경로입니다.

표준 데이터를 생성하거나 수집하고, 이를 워크스페이스에 저장하고, 조사 인터페이스를 통해 다시 읽고, 이를 리포트나 비교 결과로 변환합니다.

## 워크스페이스 (Workspace)

워크스페이스는 `Contexta`가 표준 제품 데이터를 보관하는 로컬 루트(root)입니다.

표준 공용 이름:

- `.contexta/`

개념적으로 워크스페이스는 제품이 다음을 보관하는 장소입니다:

- 메타데이터 및 관계
- 레코드
- 아티팩트
- 리포트 및 내보내기
- 복구 상태

제품을 사용하기 위해 모든 내부 파일 경로를 외울 필요는 없습니다. 중요한 것은 워크스페이스가 표준 저장 데이터의 로컬 홈이라는 점입니다.

## 프로젝트 (Project)

프로젝트는 관련 실행(run)들을 묶는 최상위 그룹입니다.

다음과 같은 것들을 그룹화하고 싶을 때 프로젝트를 사용하세요:

- 하나의 실험군 (experiment family)
- 하나의 훈련/평가 시스템
- 하나의 제품 워크플로
- 하나의 논리적 애플리케이션 경계

실제로 실행들이 명확하게 명명된 프로젝트에 속해 있을 때 많은 쿼리 및 리포팅 워크플로가 더 쉬워집니다.

## 실행 (Run)

실행(Run)은 조사의 기본 단위입니다.

하나의 실행을 나중에 다음과 같이 하고 싶은 하나의 실행 인스턴스로 생각할 수 있습니다:

- 조사 (inspect)
- 비교 (compare)
- 진단 (diagnose)
- 리포트 생성

대부분의 상위 수준 읽기 워크플로가 실행을 기준으로 고정되어 있기 때문에 실행은 시스템의 중심이 됩니다.

## 스테이지 (Stage)

스테이지는 실행의 명명된 일부입니다.

일반적인 스테이지 예시는 다음과 같습니다:

- 준비 (prepare)
- 훈련 (train)
- 평가 (evaluate)
- 내보내기 (export)

스테이지는 실행의 내부 구조를 조직화하여 레코드, 타이밍 및 아티팩트가 문맥에 맞게 해석될 수 있도록 돕습니다.

## 오퍼레이션 (Operation)

오퍼레이션은 스테이지 내부의 더 세밀한 단위입니다.

다음과 같은 경우에 오퍼레이션이 필요할 수 있습니다:

- 더 상세한 추적 (tracing)
- 더 좁은 범위의 근거 연결 (evidence links)
- 하위 단계에 대한 더 정밀한 조사

모든 사용자가 즉시 오퍼레이션을 고려할 필요는 없지만, 오퍼레이션은 제품 핵심 모델의 일부입니다.

## 레코드 (Records)

레코드는 실행과 연관된 추가 전용(append-style) 관측 엔트리이며, 종종 스테이지나 오퍼레이션과도 연관됩니다.

주요 레코드 제품군은 다음과 같습니다:

- 이벤트 (events)
- 메트릭 (metrics)
- 스팬 (spans)
- 기능 저하(Degraded) 마커

나중에 재생, 스캔, 쿼리 또는 진단하고 싶은 시간 순서상의 관측 사실들을 위해 레코드를 사용하세요.

### 이벤트 (Events)

이벤트는 발생한 일을 설명합니다.

예시:

- 데이터셋 로드됨 (dataset loaded)
- 체크포인트 저장됨 (checkpoint saved)
- 폴백(fallback) 사용됨
- 검증 실패함

### 메트릭 (Metrics)

메트릭은 측정된 값을 설명합니다.

예시:

- 정확도 (accuracy)
- 손실 (loss)
- 소요 시간 (duration)
- 아티팩트 크기

### 스팬 (Spans)

스팬은 시간이 기록된 실행 세그먼트를 설명합니다.

예시:

- 하나의 오퍼레이션 소요 시간
- 하나의 추론(inference) 호출
- 하나의 검색(retrieval) 단계

### 기능 저하(Degraded) 마커

기능 저하(Degraded) 마커는 시스템이 불완전하거나 부분적인 수집만 일어난 상태에 대해 정직하게 기록하기 위해 존재합니다.

예시:

- 부분적인 캡처 (partial capture)
- 입력값 누락
- 재생 공백 (replay gaps)
- 유입된 데이터 손실

`Contexta`는 오류를 삼긴 채 겉보기에만 정상인 상태로 두기보다 문제가 발생한 지점을 명시적으로 드러내는 것을 더 중요하게 여기기 때문에 이 마커는 시스템에서 핵심 역할을 합니다.

## 배치 (Batch)

배치 실행은 스테이지 내의 하나의 독립적인 데이터 처리 단위입니다.

예시:

- 학습 루프의 한 에포크(epoch)
- 스트리밍 파이프라인의 한 청크(chunk)
- 배치 임포트 워크플로우의 한 파일

배치는 `Run → Stage → Batch` 계층 구조에 속합니다. 스테이지 내에서
더 세밀한 진행 추적이 필요할 때 사용하세요.

자세한 내용은 [배치, 샘플 & 배포](./batch-sample-deployment.md)를 참고하세요.

## 샘플 (Sample)

샘플 관찰은 스테이지 처리 중 발견된 하나의 개별 항목을 기록합니다.

예시:

- 검증 과정에서의 한 입력 행
- 데이터셋 스캔에서의 한 이미지
- 추론 배치에서의 한 예측값

샘플은 항목별 증거가 필요할 때 유용합니다 — 예를 들어, 어떤 특정 입력이
기능 저하 상태(degradation)를 유발했는지 식별하는 경우.

## 배포 (Deployment)

배포 실행은 모델 또는 아티팩트가 환경에 제공되는 하나의 인스턴스를 추적합니다.

예시:

- 서빙 엔드포인트에 푸시된 모델
- 스테이징으로 승격된 체크포인트
- 모델 레지스트리에 등록된 아티팩트

배포는 프로젝트에 범위가 지정되며, 해당 아티팩트를 생성한 런과 연결할 수 있습니다.

## 아티팩트 (Artifacts)

아티팩트는 실행 및 스테이지와 연관된 저장된 출력물 또는 파일입니다.

예시:

- 체크포인트 (checkpoints)
- 생성된 리포트
- 내보낸 번들 (exported bundles)
- 중간 모델 출력물

아티팩트는 단순한 파일 경로가 아닙니다. 표준 저장 데이터의 일부이며 검증, 리니지, 가져오기/내보내기 및 리포팅 워크플로에 참여할 수 있습니다.

## 리니지 (Lineage)

리니지는 엔티티들이 어떻게 연결되어 있는지를 설명합니다.

다음과 같은 질문에 답하고 싶을 때 리니지를 사용하세요:

- 이 아티팩트는 어디서 왔는가?
- 어떤 실행이 이 결과를 생성했는가?
- 이 대상의 상위(upstream) 또는 하위(downstream)에는 무엇이 있는가?

리니지는 특히 파이프라인을 디버깅하거나 결과가 어떻게 도출되었는지 설명할 때 유용합니다.

## 프로버넌스 (Provenance)

프로버넌스(Provenance)는 도출된 결과를 신뢰하고 나중에 다시 재현할 수 있도록 돕는 맥락적 출처 정보를 뜻합니다.

예시:

- 환경 정보
- 코드 리비전(revision) 맥락
- 실행 메타데이터
- 관계 이력

리니지가 각 출처와 대상 요소들이 어떻게 연결되어 있는지 알려준다면, 프로버넌스는 어떤 조건과 맥락에서 그 현상이 일어났는지 설명하는 데 도움을 줍니다.

## 쿼리 (Query)

쿼리는 표준 저장 데이터를 검색하기 위한 읽기 중심의 인터페이스입니다.

일반적인 쿼리 동작은 다음과 같습니다:

- 실행 목록 조회
- 단일 실행 스냅샷 가져오기
- 관련 아티팩트 찾기
- 스테이지 및 레코드 증거 조사

쿼리는 저장 이후의 첫 번째 조사 계층입니다.

## 비교 (Compare)

비교는 다음 질문에 답합니다:

- 하나의 실행이 다른 실행과 어떻게 다른가?

다음 사항들을 조사하고 싶을 때 비교 기능을 사용하세요:

- 메트릭의 변화
- 구조적 차이점
- 리포트의 차이점
- 여러 실행에 걸친 결과 변화

## 진단 (Diagnostics)

진단은 문제 및 조사 힌트에 대한 제품의 해석 계층입니다.

시스템이 다음 사항들을 알려주기를 원할 때 진단 기능을 사용하세요:

- 무엇이 부분적 수집 혹은 저하(degraded)된 것으로 보이는가
- 무엇이 의심스러워 보이는가
- 어떤 이슈를 추가로 조사할 가치가 있는가

진단은 실패에 관한 것뿐만 아니라 숨겨진 문제를 시각화하는 것에 관한 것이기도 합니다.

## 리포트 (Report)

리포트는 표준 저장 데이터로부터 빌드된 구조화된 파생 문서입니다.

리포트를 통해 실행 또는 비교 데이터를 다음과 같이 하기 쉬운 형태로 바꿀 수 있습니다:

- 검토 (review)
- 공유
- 보관 (archive)
- 원시 스토어 외부에서 조사

리포트는 표준 저장 데이터의 하위 스트림이며, 이를 대체하는 것이 아닙니다.

## 복구 (Recovery)

복구는 데이터를 안전하게 보존, 재생, 복구 또는 마이그레이션하는 데 도움이 되는 워크플로를 다룹니다.

예시:

- 재생 (replay)
- 백업 (backup)
- 복원 (restore)

관측성 데이터는 안전하게 복구하고 이동할 수 있을 때만 유용하기 때문에 복구는 제품의 일부입니다.

## 안정(Stable), 고급(Advanced), 내부(Internal)

문서에서 이 라벨들을 반복적으로 보게 될 것입니다.

### 안정 (Stable)

새로운 코드와 문서를 위해 권장되는 공용 계약입니다.

### 고급 (Advanced)

공용 인터페이스이지만, 주로 파워 유저와 운영자를 위한 것입니다.

### 내부 (Internal)

구현 세부 사항이며, 지원되는 공용 임포트 대상이 아닙니다.

## 멘탈 모델 요약

짧은 멘탈 모델 하나만 기억하고 싶다면 다음을 참고하세요:

- **워크스페이스**는 표준 데이터를 보관합니다.
- **프로젝트**는 관련 실행들을 그룹화합니다.
- **실행(Run)**은 조사의 주요 단위입니다.
- **스테이지**는 실행 구조를 조직화합니다.
- **레코드**는 관측 가능한 이벤트, 메트릭, 스팬 및 기능 저하(degraded) 상태를 캡처합니다.
- **아티팩트**는 저장된 출력물과 파일을 나타냅니다.
- **리니지**와 **프로버넌스**는 결과가 어떻게 연결되고 어디서 왔는지 설명합니다.
- **쿼리**, **비교**, **진단** 및 **리포트**는 저장된 데이터를 조사 결과로 변환합니다.
- **복구**는 실패와 변화 도중에도 시스템이 작동할 수 있게 유지합니다.
